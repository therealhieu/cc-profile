# cc-profile Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a Rust `cc-profile` CLI that stores Claude Code profiles in `~/.cc-profile`, lets users manage them interactively and through core commands, and launches `claude` with the active profile's environment and global arguments.

**Architecture:** Keep prompt/UI and `clap` parsing at the outer edge. Domain services mutate in-memory config, `ConfigRepository` owns TOML persistence and file permissions, and `LaunchService` builds a testable `CommandSpec` before executing `std::process::Command`.

**Tech Stack:** Rust 2021, `anyhow`, `bon`, `clap`, `dialoguer`, `dirs`, `serde`, `toml`, `assert_cmd`, `assert_fs`, `predicates`, `cargo-nextest`.

## Cross-Part Dependencies

```text
Part 1 config types + repository
  ↓
Part 2 services depend on Config/Profile/Args and validation
  ↓
Part 3 CLI handlers depend on repository + services
  ↓
Part 4 interactive UI depends on repository + services + launch
```

- Parts are sequential because each later part consumes public interfaces produced by earlier parts.
- Within Part 1, Tasks 1–4 are sequential; Task 5 final verification runs after Tasks 1–4.
- Within Part 2, Tasks 1–3 can run in parallel after Part 1; Task 4 depends on `CommandSpec` and runs after them.
- Within Part 3, Tasks 1–4 are sequential because each expands command dispatch and integration coverage.
- Within Part 4, Tasks 1–3 are sequential; Task 4 final verification runs last.
- If one parallel task fails and another succeeds, keep the successful commit, fix the failing task independently, then re-run its review loop.

## Execution Rules

- Use TDD for every task: failing test → expected failure → minimal implementation → passing test → commit.
- `implementer` and `tester` are sequential for each task.
- `spec-reviewer` and `code-quality-reviewer` run in parallel after implementation and testing.
- Reviewer loops stop after 2 iterations, on stalled findings, or if Critical + Important findings fail to decrease.
- Use repository git identity; do not pass `--author` or override git identity.
- Commit each task with a focused Conventional Commit under 72 characters.
- Do not log API keys. Display config views show API keys as stored because the design requires it.
- Do not implement encrypted storage, profile-specific env vars, profile-specific args, or `ANTHROPIC_AUTH_TOKEN` in v1.
